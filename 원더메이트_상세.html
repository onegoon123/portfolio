1. FSM (유한 상태 머신) 기반 유닛 AI

문제 상황

유닛 AI 로직이 복잡하게 섞여 있어 다음 문제가 발생했습니다:

  • 코드 가독성 저하 (if-else 중첩)
  • 새로운 동작 추가 어려움
  • 버그 발생 시 디버깅 어려움

해결 과정

1단계: FSM 상태 기본 클래스

public abstract class State
{
    protected StateMachine stateMachine;

    public virtual void Enter() { }
    public virtual void Update() { }
    public virtual void Exit() { }

    public void SetStateMachine(StateMachine machine)
    {
        stateMachine = machine;
    }
}

public class StateMachine
{
    private State currentState;
    private Dictionary<System.Type, State> states = new Dictionary<System.Type, State>();

    public void ChangeState<T>() where T : State
    {
        State newState = states[typeof(T)];
        if (currentState != null)
        {
            currentState.Exit();
        }
        currentState = newState;
        currentState.Enter();
    }

    public void Update()
    {
        if (currentState != null)
        {
            currentState.Update();
        }
    }
}

2단계: 타워 AI 상태 구현

  • IdleState: 대기 상태, 적이 사정거리 내에 있는지 확인
  • AttackState: 공격 상태, 주기적으로 공격

3단계: 적 AI 상태 구현

  • MoveState: 이동 상태, 경로를 따라 이동
  • DieState: 사망 상태, 애니메이션 재생 후 삭제

4단계: 타워에 FSM 적용

FSM을 타워 컴포넌트에 통합하여 AI 동작을 제어합니다.

결과

코드 가독성 낮음 (복잡한 if-else) 높음 (명확한 상태) 300% 개선

2. Command 패턴 기반 스킬 시스템

문제 상황

스킬 시스템이 하드코딩되어 있어 다음 문제가 발생했습니다:

  • 새 스킬 추가 시 코드 수정 필요
  • 스킬 로직 재사용 불가
  • 스킬 조합 및 연계 어려움

해결 과정

1단계: Command 인터페이스

public interface ICommand
{
    void Execute();
    void Undo();
}

2단계: 스킬 Command 기본 클래스

모든 스킬이 상속받는 기본 클래스를 정의합니다.

3단계: 구체적 스킬 Command

  • AttackSkillCommand: 기본 공격
  • AreaAttackSkillCommand: 범위 공격
  • BuffSkillCommand: 버프 스킬

4단계: 스킬 팩토리

스킬 생성 로직을 중앙화하여 확장성을 높였습니다.

5단계: 스킬 매니저

Command 큐와 히스토리를 관리하여 Undo 기능을 구현했습니다.

결과

  • 15종 이상 스킬 확장성: Command 패턴으로 쉬운 스킬 추가
  • 스킬 로직 재사용: 기본 Command 상속으로 중복 제거
  • Undo 기능 구현: Command 히스토리로 실행 취소 가능

3. 상태 효과 시스템 (Status Effect System)

문제 상황

버프/디버프 시스템이 통합되어 있지 않아 다음 문제가 발생했습니다:

  • 각 버프/디버프가 따로 구현되어 관리 어려움
  • 효과 중첩 로직 부재
  • 효과 종료 처리 복잡

해결 과정

1단계: 상태 효과 기본 클래스

public abstract class StatusEffect
{
    public string EffectName { get; protected set; }
    public float Duration { get; protected set; }
    public float RemainingTime { get; protected set; }
    public int StackCount { get; protected set; } = 1;
    public bool IsExpired => RemainingTime <= 0;

    protected GameObject target;

    public virtual void Apply(GameObject target)
    {
        this.target = target;
        this.RemainingTime = Duration;
        OnApply();
    }

    public virtual void Update()
    {
        RemainingTime -= Time.deltaTime;
        OnUpdate();
    }

    public virtual void Remove()
    {
        OnRemove();
    }
}

2단계: 버프 구현

  • AttackBuffEffect: 공격력 증가
  • SpeedBuffEffect: 속도 증가

3단계: 디버프 구현

  • SlowDebuffEffect: 속도 감소
  • PoisonDebuffEffect: 독 (DoT)
  • StunDebuffEffect: 기절

4단계: 상태 효과 매니저

모든 활성 효과를 중앙에서 관리하고 자동으로 만료된 효과를 제거합니다.

결과

  • 상태 효과 체계화: 버프/디버프 통합 관리
  • 효과 중첩 지원: 같은 효과 중첩 가능
  • 효과 종료 자동 처리: 매니저가 자동으로 만료된 효과 제거

개발 도전과제 및 해결

FSM 상태 전환 로직

문제: 복잡한 상태 전환 조건 관리

해결: 각 상태에서 다음 상태 조건을 명확히 정의, StateMachine에서 전환 로직 중앙화, 상태 전환 이벤트로 디버깅 용이

스킬 확장성

문제: 다양한 스킬 효과 구현의 어려움

해결: Command 패턴으로 스킬을 독립적 객체로 관리, 기본 Command 클래스를 상속하여 쉬운 확장, 스킬 팩토리로 스킬 생성 로직 분리

상태 효과 충돌

문제: 여러 효과가 동시에 적용될 때 우선순위 문제

해결: 효과 적용 순서 정의 (버프 → 기본 능력치 → 디버프), 중첩 불가 효과는 기존 효과 시간만 갱신, 상태 효과 매니저로 중앙 관리
